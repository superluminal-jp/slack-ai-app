# Research: DynamoDB Agent Registry Migration

**Feature**: 055-dynamodb-agent-registry
**Date**: 2026-03-25

## Decision 1: DynamoDB Table Schema

**Decision**: PK=`env` (String), SK=`agent_id` (String). Attributes: `arn`, `description`, `skills`, `updated_at`.

**Rationale**:
- Single Query on PK=`env` retrieves all agents for a given environment in one API call (vs S3's ListObjectsV2 + N×GetObject)
- PK-based env isolation (dev/prod) is natural for DynamoDB and matches the current S3 prefix pattern (`{env}/agent-registry/`)
- No GSI needed — the only access pattern is "get all agents for env"
- `skills` stored as List of Map (native DynamoDB type), avoiding JSON string serialization

**Alternatives considered**:
- Single-item design (one item with all agents as a nested map): Rejected — exceeds 400KB limit if agents grow, no atomic per-agent updates
- PK=`agent_id`, SK=`env`: Rejected — requires Scan to get all agents, not Query

## Decision 2: CDK Construct Pattern

**Decision**: Create `AgentRegistryTable` construct following the `WhitelistConfig` pattern exactly.

**Rationale**:
- Project has 6 existing DynamoDB table constructs, all following identical pattern: `Construct` subclass, `public readonly table`, `PAY_PER_REQUEST`, `AWS_MANAGED` encryption, `DESTROY` removal policy
- Table name: `${stackName}-agent-registry` (consistent with all other tables)
- No TTL needed — entries are overwritten at deploy time, not expired
- No PITR needed — data is regenerated by deploy scripts

**Alternatives considered**:
- Reuse an existing table with a new partition: Rejected — violates single-purpose table design, harder to manage IAM

## Decision 3: IAM Permission Pattern

**Decision**: Grant `dynamodb:Query` on the agent-registry table to the verification agent execution role. Remove S3 `grantRead` for the agent-registry bucket.

**Rationale**:
- `table.grantReadData(role)` in CDK grants `dynamodb:BatchGetItem`, `dynamodb:GetItem`, `dynamodb:Query`, `dynamodb:Scan` — acceptable for this use case
- S3 read permissions (`s3:GetObject*`, `s3:GetBucket*`, `s3:List*`) on the registry bucket are no longer needed
- Deploy scripts use their own IAM (AWS CLI credentials), not the verification agent's role, so `dynamodb:PutItem` is handled separately

**Alternatives considered**:
- Fine-grained `dynamodb:Query` only via custom policy: Rejected — `grantReadData` is the CDK standard pattern and the extra permissions (GetItem, Scan) are harmless

## Decision 4: Deploy Script Registration

**Decision**: Replace `register_agent_in_s3()` with `register_agent_in_dynamodb()` using `aws dynamodb put-item` CLI command.

**Rationale**:
- The AWS CLI `put-item` command is the simplest way to write a single DynamoDB item from bash
- Table name lookup follows the same pattern as bucket name: check env var first, fall back to CloudFormation output
- The new output key will be `AgentRegistryTableName` (replacing `AgentRegistryBucketName`)
- PutItem is an upsert — re-deploying the same agent overwrites the entry (same behavior as S3 PutObject)

**Alternatives considered**:
- Python script for registration: Rejected — adds dependency; bash + `aws dynamodb put-item` is sufficient
- Batch write for all agents: Rejected — each agent deploys independently, writes its own entry

## Decision 5: Python Module Changes

**Decision**: Replace `_load_from_s3()` with `_load_from_dynamodb()`. Change env vars from `AGENT_REGISTRY_BUCKET`+`AGENT_REGISTRY_KEY_PREFIX` to `AGENT_REGISTRY_TABLE`+`DEPLOYMENT_ENV`.

**Rationale**:
- Single `dynamodb.query(TableName, KeyConditionExpression="env = :e")` replaces ListObjectsV2 + N×GetObject
- `DEPLOYMENT_ENV` is already available in the runtime environment (used for other purposes) — but currently not passed to the AgentCore Runtime env vars. We'll need to add it.
- Actually: looking at the current code, `AGENT_REGISTRY_KEY_PREFIX` already encodes the env (e.g., `dev/agent-registry/`). For DynamoDB, we need the raw env value. We can either: (a) add `DEPLOYMENT_ENV` as a new env var, or (b) pass it as `AGENT_REGISTRY_ENV`. Option (b) is cleaner — dedicated env var for registry env.
- Final decision: Use `AGENT_REGISTRY_TABLE` (table name) + `AGENT_REGISTRY_ENV` (partition key value, e.g., "dev"). This keeps the agent_registry module self-contained without depending on a global `DEPLOYMENT_ENV`.

**Alternatives considered**:
- Parse env from table name (e.g., `SlackAI-Verification-Dev-agent-registry` → "dev"): Rejected — fragile, couples to naming convention
- Use `DEPLOYMENT_ENV` directly: Acceptable, but `AGENT_REGISTRY_ENV` is more explicit and doesn't assume the global var exists
