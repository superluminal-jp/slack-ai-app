# Feature Specification: Usage History Data Archive Replication

**Feature Branch**: `041-s3-replication-archive`
**Created**: 2026-03-16
**Status**: Draft

## User Scenarios & Testing *(mandatory)*

### User Story 1 — 利用履歴データの自動アーカイブ (Priority: P1)

オペレーションチームは、Verification Agent が記録した会話テキスト（入力・出力）、添付ファイル、DynamoDB テーブルエクスポートが、プライマリバケットへの書き込みと同時に別の独立したアーカイブバケットへ自動的に複製されることを求めている。これにより、プライマリバケットで誤削除・データ破損・設定ミスが発生しても、アーカイブから完全なデータを復元できる。

**Why this priority**: プライマリバケットへの書き込みは既に機能している。アーカイブが存在しない間は、削除・破損イベントが回復不能なデータ損失に直結するため最優先。

**Independent Test**: アーカイブバケットを確認し、プライマリバケットの全プレフィックス（`content/`、`attachments/`、`dynamodb-exports/`）のオブジェクトが複製されていること。

**Acceptance Scenarios**:

1. **Given** 会話が発生し入力テキストがプライマリバケットの `content/` プレフィックスに書き込まれた、**When** 一定時間経過後、**Then** 同一オブジェクトがアーカイブバケットの `content/` プレフィックスに存在する。
2. **Given** Slack ファイルが添付されプライマリバケットの `attachments/` に保存された、**When** 一定時間経過後、**Then** アーカイブバケットの `attachments/` に同一ファイルが存在する。
3. **Given** DynamoDB 日次エクスポートがプライマリバケットの `dynamodb-exports/` に完了した、**When** 一定時間経過後、**Then** アーカイブバケットの `dynamodb-exports/` に全エクスポートファイルが存在する。
4. **Given** プライマリバケット上のオブジェクトが誤って削除された、**When** アーカイブバケットを確認すると、**Then** そのオブジェクトはアーカイブバケットに残っている。

### User Story 2 — クロスアカウント移行対応 (Priority: P2)

オペレーションチームは、将来アーカイブ先を専用のセキュリティ・コンプライアンス用 AWS アカウントへ移行する可能性に備え、コード変更なしに宛先アカウント ID と IAM 情報だけを設定変更してアーカイブ先を切り替えられる構成を求めている。

**Why this priority**: 現時点では同一アカウント実装で十分だが、設計段階でクロスアカウントを想定しておくことで将来の移行コストをゼロに近づける。P1 完了後に対応可能。

**Independent Test**: アーカイブ先アカウント ID および IAM プリンシパルを設定値として外部から注入できること。設定変更のみでクロスアカウント構成に切り替えられることをアーキテクチャレビューで確認。

**Acceptance Scenarios**:

1. **Given** アーカイブ先として別 AWS アカウントを指定する設定値が存在する、**When** その設定値のみを変更して再デプロイする、**Then** コードの変更なしに別アカウントのバケットへの複製が成立する。
2. **Given** 同一アカウントでアーカイブが稼働中、**When** 設定値に宛先アカウント ID を追加して再デプロイする、**Then** クロスアカウント複製に切り替わり、以降の新規オブジェクトが別アカウントバケットに複製される。

---

### Edge Cases

- プライマリバケットへのオブジェクト書き込みが高頻度で発生した場合、アーカイブへの複製に遅延が生じても、プライマリバケットへの書き込み自体はブロックされない。
- アーカイブバケット側で障害が発生した場合、プライマリバケットへの書き込みには影響しない。
- レプリケーション有効化以前に存在するオブジェクト（既存データ）は自動複製の対象外。必要であれば別途バックフィル手段を講じる（本仕様スコープ外）。
- アーカイブバケットでオブジェクトが削除された場合、プライマリバケットのオブジェクトは影響を受けない。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: プライマリ使用履歴バケットに書き込まれた全オブジェクト（`content/`、`attachments/`、`dynamodb-exports/` のすべてのプレフィックス）は、書き込み後自動的にアーカイブバケットへ複製されなければならない。
- **FR-002**: アーカイブバケットはプライマリバケットとは独立した別リソースでなければならない（同一アカウント・同一リージョン実装）。
- **FR-003**: アーカイブバケットは、プライマリバケットと同等のセキュリティ設定（暗号化、SSL 強制、パブリックアクセス全面ブロック）を維持しなければならない。
- **FR-004**: アーカイブバケットのオブジェクトも 90 日後に自動削除されなければならない（プライマリと同一保存期間）。
- **FR-005**: プライマリバケットへの書き込み処理は、アーカイブ複製の成否に依存してはならない（複製失敗はエージェントの応答に影響しない）。
- **FR-006**: アーカイブ先の AWS アカウント ID は外部設定値として注入可能でなければならない（将来のクロスアカウント移行をコード変更なしに実現するため）。
- **FR-007**: レプリケーション構成には、アーカイブバケット側で宛先アカウントによるオブジェクト所有権を確立できるパーミッション設計が含まれなければならない。

### Key Entities

- **プライマリ使用履歴バケット**: 会話データ・添付ファイル・DynamoDB エクスポートを保持する既存バケット。本仕様でレプリケーション送信元となる。
- **アーカイブバケット**: 本仕様で新規作成される独立した複製先バケット。プライマリバケットと同一リージョン・同一（または別）アカウントに配置。
- **レプリケーション設定**: プライマリバケットに付与する「新規オブジェクトを自動複製する」設定。対象プレフィックスと宛先バケットを定義する。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: プライマリバケットへの新規書き込み後 15 分以内に、同一オブジェクトがアーカイブバケットに存在する。
- **SC-002**: 複製されたオブジェクトのコンテンツがプライマリバケットのオリジナルと完全に一致する（バイト一致）。
- **SC-003**: アーカイブ宛先を別 AWS アカウントへ変更する際に必要なコード変更がゼロである（設定値の変更のみ）。
- **SC-004**: プライマリバケットへの書き込みレイテンシが、レプリケーション有効化の前後で変化しない。
- **SC-005**: アーカイブバケットのオブジェクトが 90 日後に自動削除される（プライマリと同一保存期間）。

## Assumptions

- アーカイブバケットは同一リージョン（`ap-northeast-1`）に配置する（同一リージョンレプリケーション）。
- レプリケーションはプロビジョニング後に書き込まれた新規オブジェクトのみ対象。既存オブジェクトのバックフィルは本仕様スコープ外。
- アーカイブバケットへのバケット名は `{stackName.toLowerCase()}-usage-history-archive` パターンとする。
- バージョニングはレプリケーションの前提条件としてプライマリバケットで有効化する。古いバージョンのオブジェクトは数日で自動削除し、ストレージコスト増加を抑制する。
- 現フェーズは同一アカウント実装。クロスアカウント対応は設定値の外部注入と IAM 設計で担保し、将来の移行はコード変更不要とする。
