# Feature Specification: Verification Zone Usage History

**Feature Branch**: `039-usage-history`
**Created**: 2026-03-16
**Status**: Draft
**Input**: User description: "検討内容を踏まえて利用履歴保存を実装"

## User Scenarios & Testing *(mandatory)*

### User Story 1 — 利用履歴の記録 (Priority: P1)

オペレーターが Verification Agent に対してリクエストを送ると、そのやり取り（入力テキスト全文・出力テキスト全文・添付ファイル・パイプライン通過結果・所要時間）が自動的に保存される。記録はリクエスト処理の成否に影響せず、保存失敗はリクエスト継続に影響しない。

**Why this priority**: 履歴が記録されなければ利用状況の把握・障害調査・コスト管理が不可能になる。他のすべてのストーリーの前提となる基盤機能。

**Independent Test**: Verification Agent に Slack メッセージを送信した後、永続ストレージにレコードが生成されていることを確認することで単独テスト可能。主要機能としての価値は「何も記録されていない状態からすべての利用が記録される状態への移行」で確認できる。

**Acceptance Scenarios**:

1. **Given** 通常の Slack テキストメッセージが送信された、**When** リクエストが処理完了した、**Then** 入力テキスト全文・出力テキスト全文・パイプライン結果・所要時間が永続ストレージに保存される
2. **Given** Slack ファイル添付付きメッセージが送信された、**When** リクエストが処理完了した、**Then** 入力テキスト・出力テキストに加え、添付ファイルが個別にファイル専用ストレージへ保存される
3. **Given** 利用履歴の保存処理が失敗した（ストレージ障害等）、**When** リクエストが処理された、**Then** エラーが警告ログに記録されるが、ユーザーへのレスポンス送信は正常に完了する
4. **Given** セキュリティパイプラインがリクエストを拒否した（ホワイトリスト外・レート超過）、**When** 拒否が発生した、**Then** リクエストメタデータ（チャンネル・ユーザー・拒否理由）が保存される（入出力テキストは存在しないため保存しない）

---

### User Story 2 — 保持期間による自動削除 (Priority: P2)

保存された利用履歴は指定された保持期間（デフォルト 90 日）を過ぎると自動的に削除される。コスト管理とデータ最小化の観点から、不要なデータを手動管理なしに廃棄する。

**Why this priority**: 保持期間管理がなければストレージコストが無限に増大し、不要データの長期保存はデータ保護上のリスクとなる。ただし P1 の記録機能が存在して初めて意味を持つ。

**Independent Test**: 保持期間を超えた古いレコードが自動的に削除されることを確認することで単独テスト可能。

**Acceptance Scenarios**:

1. **Given** 保持期間 90 日を超えたレコードが存在する、**When** ストレージの自動クリーンアップが実行された、**Then** 該当レコードおよびファイルが削除される
2. **Given** 保持期間内のレコードが存在する、**When** ストレージの自動クリーンアップが実行された、**Then** 該当レコードは削除されない

---

### User Story 3 — 相関 ID による履歴の参照 (Priority: P3)

オペレーターが `correlation_id` を指定して特定のリクエストの詳細（入出力テキスト・添付ファイル一覧・パイプライン結果）を取得できる。障害調査やデバッグに使用する。

**Why this priority**: P1・P2 が揃って初めて参照が意味を持つ。現時点では CloudWatch Logs に同一 `correlation_id` でログが残るため、参照機能がなくても最低限の調査は可能。

**Independent Test**: 既知の `correlation_id` を使って履歴レコードを検索し、正しいリクエスト内容が返ることを確認することで単独テスト可能。

**Acceptance Scenarios**:

1. **Given** `correlation_id` が既知のレコードが保存されている、**When** `correlation_id` でレコードを検索した、**Then** 対応する入出力テキストおよびメタデータが返される
2. **Given** 存在しない `correlation_id` を指定した、**When** 検索を実行した、**Then** レコードが存在しないことを示す結果が返される

---

### Edge Cases

- 入力テキストまたは出力テキストが空文字・null の場合は空文字として保存する
- 添付ファイルが複数枚ある場合、各ファイルを個別エントリとして保存し `correlation_id` で紐付ける
- 添付ファイルのサイズが上限（10 MB）を超えている場合、そのファイルは保存をスキップしメタデータにスキップ理由を記録する
- ストレージへの書き込みがタイムアウトした場合、リトライは行わずログに警告を記録して継続する
- `correlation_id` が存在しない（異常系）場合は UUID を生成して代替し、異常を警告ログに残す

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは、Verification Agent へのリクエスト処理完了後に、入力テキスト全文・出力テキスト全文・リクエストメタデータをテキスト用永続ストレージに保存しなければならない
- **FR-002**: システムは、Slack 添付ファイルが存在する場合に、各ファイルをファイル専用永続ストレージに保存しなければならない
- **FR-003**: システムは、セキュリティパイプラインがリクエストを拒否した場合も、メタデータ（チャンネル ID・ユーザー ID・拒否理由・タイムスタンプ）を保存しなければならない
- **FR-004**: システムは、利用履歴の保存が失敗した場合でも、ユーザーへのレスポンス送信を正常に完了させなければならない（保存失敗がサービス停止を引き起こしてはならない）
- **FR-005**: 各テキストレコードは保持期間（デフォルト 90 日）を超えると自動的に削除されなければならない
- **FR-006**: 各ファイルは保持期間（デフォルト 90 日）を超えると自動的に削除されなければならない
- **FR-007**: 各レコードは `correlation_id` で一意に検索できなければならない
- **FR-008**: チャンネル ID および日時による範囲クエリができなければならない
- **FR-009**: 添付ファイルとテキストレコードは同一の `correlation_id` で紐付けられなければならない
- **FR-010**: システムは、添付ファイルの保存をスキップした場合、スキップしたファイルの識別子と理由をメタデータに記録しなければならない

### Key Entities

- **UsageRecord**: 1リクエストに対応する利用履歴の中心エンティティ。`correlation_id`（一意識別子）、`channel_id`（Slack チャンネル）、`team_id`・`user_id`（送信者）、`created_at`（タイムスタンプ）、`input_text`（ユーザー入力全文）、`output_text`（エージェント応答全文）、パイプライン通過フラグ群（existence check / authorization / rate limit）、オーケストレーション結果（呼び出したエージェント一覧・ターン数・成否）、`duration_ms`（所要時間）、ファイル参照一覧、保持期限を持つ
- **AttachmentRecord**: Slack 添付ファイルの永続化エントリ。`correlation_id` で UsageRecord に紐付く。ファイル名、ファイルサイズ、保存先識別子、保存タイムスタンプ、保持期限を持つ
- **PipelineResult**: セキュリティパイプラインの各ステージ通過結果をまとめた値オブジェクト。拒否された場合はステージ名と拒否理由を含む

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 正常完了リクエストの 99.9% 以上で利用履歴レコードが作成される（記録漏れが 0.1% 未満）
- **SC-002**: 利用履歴の保存処理によるリクエスト全体のレイテンシ増加が 50 ms 以内に収まる
- **SC-003**: 保存失敗（ストレージ障害等）によってユーザーへのレスポンスが失敗する件数がゼロである
- **SC-004**: 保持期間（90 日）を超えたレコードが 24 時間以内に自動削除される
- **SC-005**: `correlation_id` による履歴検索が 500 ms 以内に結果を返す

## Assumptions

- **機密性保持**: 入出力テキスト（会話内容）はインデックス用ストレージに格納せず、ファイルストレージに分離して保存する。インデックス用ストレージはメタデータのみを保持する
- 入出力テキストのサイズ制限はファイルストレージの制約に従う（実運用上の上限なし）
- 添付ファイルの保存先として、既存の一時ファイル交換用ストレージとは別に、長期保存専用ストレージを使用する
- 保持期間 90 日はシステム全体で統一する（チャンネル・ユーザー単位の個別設定は対象外）
- 利用履歴への読み取りアクセスは管理用途（障害調査・監査）に限定し、エンドユーザーへの自動公開は対象外
- 既存の `correlation_id` ライフサイクル（パイプライン全体で伝播）は変更しない
- Slack 添付ファイルは既にパイプライン内で一時ストレージにアップロード済みであるため、コピー操作のみで永続化できる

## Dependencies

- 既存の Verification Agent パイプラインに統合する
- 既存の一時ファイル交換ストレージからのコピー操作で添付ファイルを永続化する
- 既存の `correlation_id` 伝播機構に依存する
- 新規インフラストラクチャ（テキスト用ストレージ・ファイル専用ストレージ・アクセス権限）の追加が必要
