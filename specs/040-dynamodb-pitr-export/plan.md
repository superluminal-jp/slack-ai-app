# Implementation Plan: DynamoDB Usage History Daily S3 Export via PITR

**Branch**: `040-dynamodb-pitr-export` | **Date**: 2026-03-16 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/040-dynamodb-pitr-export/spec.md`

## Summary

Enable PITR on the existing `UsageHistoryTable`, add a `dynamodb-exports/` lifecycle rule to `UsageHistoryBucket`, and introduce a `DynamoDbExportJob` CDK construct that schedules a daily Python Lambda (EventBridge cron, UTC 00:00) to initiate a native DynamoDB-to-S3 full export. All changes are CDK/infrastructure only — no `src/` Python agent code changes required.

**Performance Goals**: Export initiation < 1s (Lambda only calls API and returns); export completion 5–60 minutes (DynamoDB service async)
**Constraints**: Least-privilege IAM; fail-open (export failure must not affect Slack responses); no new Python packages

## Architecture

```
EventBridge Scheduler (cron: 0 15 * * ? *, UTC 15:00 = JST 00:00 daily)
        │
        ▼
DynamoDbExportJob Lambda (Python 3.11)
  ├── Reads: TABLE_ARN, EXPORT_BUCKET_NAME env vars
  ├── Calls: dynamodb:ExportTableToPointInTime
  └── Returns: {status: "export_initiated", export_arn: "..."}
        │ (async — Lambda returns immediately)
        ▼
DynamoDB Service (async export execution)
  └── Writes: s3://usage-history-bucket/dynamodb-exports/{YYYY/MM/DD}/{export-id}/

CloudWatch Alarm: Lambda Errors > 0 in 5 minutes
```

## Key Design Decisions

### 1. EventBridge Scheduler (Stable L2 in aws-cdk-lib 2.215.0)

`aws-cdk-lib/aws-scheduler` and `aws-cdk-lib/aws-scheduler-targets` are stable in `aws-cdk-lib` 2.215.0 — no alpha package needed. EventBridge Scheduler is the preferred modern scheduling service over EventBridge Rules (supports retry policies, DLQ, flexible time windows). See `research.md`.

### 2. Lambda-initiated Native DynamoDB Export

DynamoDB Export to S3 is fully managed: handles large tables, parallel writes, no timeout risk. Lambda only initiates (< 1s). Alternative (Lambda reading/writing data) is rejected: complex, expensive, timeout-prone.

### 3. Lambda Role Holds All S3 Permissions (No DynamoDB Service Principal)

`ExportTableToPointInTime` uses the **calling Lambda role's permissions** to write to S3. There is no `dynamodb.amazonaws.com` service principal for this API. Lambda role needs: `dynamodb:ExportTableToPointInTime` + `s3:PutObject` + `s3:AbortMultipartUpload` + `s3:PutObjectAcl` on `dynamodb-exports/*`. No bucket policy change needed for same-account exports.

### 4. No New Python Agent `src/` Changes

This feature is pure infrastructure. No changes to `src/pipeline.py`, `src/usage_history.py`, or other agent source files. All TDD applies only to CDK (TypeScript Jest tests).

## IAM Design (Least Privilege)

### DynamoDbExportJob Lambda Role

```
dynamodb:ExportTableToPointInTime → table ARN only
s3:PutObject                      → bucket ARN/dynamodb-exports/*
s3:AbortMultipartUpload           → bucket ARN/dynamodb-exports/*
s3:PutObjectAcl                   → bucket ARN/dynamodb-exports/*
```

No bucket policy change needed. Lambda role holds all required permissions for same-account exports.

### DynamoDbExportJob Lambda Handler (`lambda/dynamodb-export-job/handler.py`)

```python
import boto3, os, json, datetime, logging

logger = logging.getLogger(__name__)

def lambda_handler(event, context):
    region = os.environ.get("AWS_REGION_NAME", "ap-northeast-1")
    ddb = boto3.client("dynamodb", region_name=region)
    bucket = os.environ["EXPORT_BUCKET_NAME"]
    table_arn = os.environ["TABLE_ARN"]
    date_path = datetime.datetime.now(datetime.timezone.utc).strftime("%Y/%m/%d")

    response = ddb.export_table_to_point_in_time(
        TableArn=table_arn,
        S3Bucket=bucket,
        S3Prefix=f"dynamodb-exports/{date_path}",
        ExportFormat="DYNAMODB_JSON",
    )
    export_arn = response["ExportDescription"]["ExportArn"]
    logger.info("Export initiated", extra={"export_arn": export_arn, "date_path": date_path})
    return {"status": "export_initiated", "export_arn": export_arn}
```
