## スライド 2: エグゼクティブサマリー

### なぜセキュリティが重要なのか？

AI システムは、機密情報を扱う可能性があり、不正アクセスから組織を保護する必要があります。

### このシステムのセキュリティアプローチ

- **多層防御**: 複数のセキュリティ層により、1 つの層が突破されても他の層が防御
- **2 鍵防御**: 2 つの独立した鍵により、1 つの鍵が漏洩しても安全
- **24 時間 365 日の監視**: 異常を即座に検出し、自動的にブロック

### ユーザーの役割

- 機密情報を含む質問を避ける
- 適切なチャンネルの使用
- 利用規約の遵守

## スライド 3: セキュリティの重要性

### 組織を守るための多層防御アプローチ

**なぜセキュリティが重要なのか？**

1. **機密情報の保護**: 組織の機密情報が漏洩するリスクを最小化
2. **不正アクセスの防止**: 外部からの攻撃や内部の不正利用を防止
3. **コンプライアンス**: 法的要件や業界基準を満たす
4. **信頼の維持**: 顧客やパートナーからの信頼を維持

**このシステムのアプローチ**:

- 単一の防御に依存せず、複数の層で防御
- 1 つの層が突破されても、他の層が防御を継続

---

## スライド 4: 多層防御の概念

### 複数のセキュリティ層により、組織を包括的に保護

**多層防御とは？**

建物のセキュリティと同じ考え方です：

- **1 層目**: エントランスの警備員（署名検証）
- **2 層目**: 受付での本人確認（実在性確認）
- **3 層目**: 各フロアへのアクセス制御（認可チェック）
- **4 層目**: 重要な部屋への追加認証（内部 API 保護）

**このシステムの 5 つの層**:

1. **Slack ユーザー認証**: SSO + MFA（Slack が提供）
2. **署名検証**: Slack からの正規リクエストのみを受け付け
3. **実在性確認**: ワークスペース、ユーザー、チャンネルが実在することを確認
4. **認可チェック**: 許可されたユーザーとチャンネルのみが利用可能
5. **内部 API 保護**: システム内部の通信を保護

## スライド 5: 第 1 層 - 署名検証

### Slack からの正規リクエストのみを受け付け、偽造リクエストを検出

**署名検証とは？**

Slack が送信するすべてのリクエストには、特別な「署名」が含まれています。この署名は、リクエストが本当に Slack から送信されたものであることを証明します。

**どのように機能するか？**

1. Slack がリクエストを送信する際に、秘密鍵を使って署名を生成
2. システムがリクエストを受信した際に、同じ秘密鍵を使って署名を検証
3. 署名が一致しない場合、リクエストを拒否

**防御する脅威**:

- 外部からの偽造リクエスト
- 古いリクエストの再利用（リプレイアタック）

**タイムスタンプ検証**: リクエストが 5 分以内に送信されたものであることを確認し、古いリクエストの再利用を防止

## スライド 6: 第 2 層 - 実在性確認（2 鍵防御の鍵 2）

### ワークスペース、ユーザー、チャンネルが実在することを確認

**実在性確認とは？**

リクエストに含まれるワークスペース ID、ユーザー ID、チャンネル ID が、実際に Slack に存在することを確認します。

**どのように機能するか？**

1. システムが Slack API を呼び出し、ワークスペース、ユーザー、チャンネルが存在するか確認
2. 存在しない場合、リクエストを拒否
3. 存在する場合、次の層に進む

**防御する脅威**:

- 偽造されたワークスペース ID、ユーザー ID、チャンネル ID を使用した攻撃
- 削除されたユーザーやチャンネルからのリクエスト

**パフォーマンス最適化**: 確認結果を 5 分間キャッシュし、同じワークスペース・ユーザー・チャンネルからの連続リクエストを高速化

## スライド 7: 2 鍵防御モデル

### 2 つの独立した鍵により、1 つの鍵が漏洩しても安全

**2 鍵防御とは？**

銀行の金庫と同じ考え方です。金庫を開けるには、2 つの鍵が必要です。1 つの鍵が盗まれても、もう 1 つの鍵がなければ金庫は開きません。

**このシステムの 2 つの鍵**:

**鍵 1: 署名シークレット**

- Slack からの正規リクエストであることを確認
- 偽造リクエストを検出

**鍵 2: ボットトークン**

- ワークスペース、ユーザー、チャンネルが実在することを確認
- 削除されたユーザーやチャンネルからのリクエストをブロック

**防御効果**: 1 つの鍵が漏洩しても、もう 1 つの鍵がなければ攻撃は成功しない

---

## スライド 8: 2 鍵防御の効果

### 1 つの鍵が漏洩しても、もう 1 つの鍵がなければ攻撃は成功しない

**攻撃シナリオ 1: 鍵 1 のみ漏洩**

- 攻撃者は正しい署名を生成可能
- しかし、実在しないワークスペース ID を使用したリクエストは実在性確認で失敗
- **結果**: 攻撃はブロックされる

**攻撃シナリオ 2: 鍵 2 のみ漏洩**

- 攻撃者は実在性確認を通過可能
- しかし、正しい署名を生成できない
- **結果**: 攻撃はブロックされる

**攻撃シナリオ 3: 両方の鍵が漏洩**

- これは正規ワークスペースの完全侵害を意味する
- しかし、後続のセキュリティ層（認可チェック、Guardrails）が防御
- **結果**: 多層防御により、攻撃の影響を最小化

**セキュリティ原則**: 防御の深層化（Defense in Depth）

## スライド 9: 第 3 層 - 認可チェック

### 許可されたユーザーとチャンネルのみが利用可能

**認可チェックとは？**

署名検証と実在性確認を通過したリクエストが、実際にシステムを利用する権限があるかを確認します。

**どのように機能するか？**

1. システムがホワイトリストを確認
2. リクエストのワークスペース、ユーザー、チャンネルがホワイトリストに含まれているか確認
3. 含まれていない場合、リクエストを拒否

**柔軟な設定**:

- ワークスペース単位での許可
- ユーザー単位での許可
- チャンネル単位での許可
- 組み合わせ可能

**防御する脅威**:

- 未認可のワークスペース、ユーザー、チャンネルからのアクセス
- 権限昇格の試行

## スライド 10: 第 4 層 - 内部 API 保護

### システム内部の通信を保護し、不正アクセスを防止

**内部 API 保護とは？**

システム内部のコンポーネント間の通信を保護し、外部からの直接アクセスを防止します。

**どのように機能するか？**

1. システム内部のコンポーネント間の通信には、特別な認証が必要
2. 外部からの直接アクセスは拒否
3. 許可されたコンポーネントのみが内部 API にアクセス可能

**防御する脅威**:

- 外部からの内部 API への直接アクセス
- システム内部のコンポーネント間の通信の傍受

**最小権限の原則**: 各コンポーネントは、必要最小限の権限のみを持つ

---

## スライド 11: 第 5 層 - AI 保護（Guardrails）

### AI の出力を監視し、不適切な内容を自動的に検出

**AI 保護（Guardrails）とは？**

AI の出力を監視し、不適切な内容（有害コンテンツ、機密情報の漏洩など）を自動的に検出します。

**どのように機能するか？**

1. AI の出力を自動的に監視
2. 不適切な内容を検出した場合、出力をブロック
3. セキュリティイベントとして記録

**防御する脅威**:

- 有害コンテンツの生成
- 機密情報の漏洩
- 不適切な内容の生成

**継続的な改善**: Guardrails の設定を継続的に改善し、新しい脅威に対応

## スライド 12: 24 時間 365 日の監視

### 異常を即座に検出し、自動的にブロック

**監視システムとは？**

システムの動作を 24 時間 365 日監視し、異常を即座に検出します。

**監視項目**:

- 不正アクセスの試行
- セキュリティイベントの発生
- システムの異常動作
- パフォーマンスの低下

**アラート**:

- 異常が検出されると、即座にセキュリティチームに通知
- 自動的にリクエストをブロック
- 詳細なログを記録

**ログ記録**:

- すべてのセキュリティイベントを記録
- 1 年間保存し、必要に応じて調査可能

## スライド 13: セキュリティイベントの例

### 実際に検出されるセキュリティイベントの例

**検出されるイベント**:

1. **署名検証失敗**

   - 偽造リクエストの試行
   - 古いリクエストの再利用
   - **対応**: 自動的にブロック

2. **実在性確認失敗**

   - 偽造されたワークスペース ID、ユーザー ID、チャンネル ID の使用
   - 削除されたユーザーやチャンネルからのリクエスト
   - **対応**: 自動的にブロック

3. **認可失敗**

   - 未認可のワークスペース、ユーザー、チャンネルからのアクセス試行
   - **対応**: 自動的にブロック

4. **異常なアクセスパターン**
   - 短時間に大量のリクエスト
   - 通常と異なるアクセスパターン
   - **対応**: 自動的にブロック、セキュリティチームに通知

**適切なチャンネルの使用**:

- 一般的な質問

- 機密情報を含む質問

- 個人情報を含む質問

**利用規約の遵守**:

1. **不適切な内容の回避**

   - 有害コンテンツを含む質問は避ける
   - 違法な内容を含む質問は避ける

2. **適切な利用**
   - システムの目的に沿った利用
   - 過度な利用の回避

## 付録 A: よくある質問（FAQ）

### Q1: 機密情報を含む質問をしても大丈夫ですか？

A: セキュリティポリシーに従い、機密情報を含む質問は避けてください。システムは多層防御により保護されていますが、ユーザーも適切に利用することが重要です。

### Q2: セキュリティイベントが発生した場合、どうなりますか？

A: システムが自動的に検出し、リクエストをブロックします。セキュリティチームに通知され、詳細な調査が行われます。

### Q3: 2 鍵防御とは何ですか？

A: 2 つの独立した鍵（署名シークレットとボットトークン）により、1 つの鍵が漏洩しても、もう 1 つの鍵がなければ攻撃は成功しない仕組みです。

### Q4: 多層防御とは何ですか？

A: 複数のセキュリティ層により、1 つの層が突破されても他の層が防御する仕組みです。建物のセキュリティと同じ考え方です。

### Q5: セキュリティインシデントが発生した場合、誰に連絡すればよいですか？

A: システム管理者またはセキュリティチームに連絡してください。緊急の場合は、即座に報告してください。

**2 鍵防御**: 2 つの独立した鍵により、1 つの鍵が漏洩しても、もう 1 つの鍵がなければ攻撃は成功しないセキュリティモデル。

**多層防御**: 複数のセキュリティ層により、1 つの層が突破されても他の層が防御するセキュリティアプローチ。

**認可チェック**: リクエストがシステムを利用する権限があるかを確認するセキュリティ機能。

**Guardrails**: AI の出力を監視し、不適切な内容を自動的に検出するセキュリティ機能。

**セキュリティイベント**: セキュリティに関する異常や脅威を表すイベント。システムが自動的に検出し、記録します。
